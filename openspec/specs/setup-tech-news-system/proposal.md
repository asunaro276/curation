# Proposal: 技術ニュース自動収集・配信システムのセットアップ

## 概要
技術記事、ライブラリのリリースノート、プレスリリースを自動収集し、Claude APIで要約してSlackに定期投稿するシステムを構築する。

## 動機
- 技術情報の収集と整理に手動で時間を費やす必要がある
- 最新の技術トレンドを継続的に追跡することが困難
- チーム内で技術情報を共有する効率的な仕組みが不足

## 目標
1. 複数の技術情報ソースから自動的にコンテンツを収集
2. LLMを活用して記事を要約し、読みやすい形式に整形
3. Slackチャンネルに定期的に投稿して情報共有を自動化
4. GitHub Actionsで完全に自動化された実行環境を構築

## 範囲内
- 基本的な技術ニュース収集機能（RSS、API対応）
- Claude APIを使用した記事要約機能
- Slack Webhook経由の投稿機能
- GitHub Actionsによるcronベースの定期実行
- 環境変数ベースの設定管理
- 基本的なエラーハンドリングとロギング

## 範囲外
- Webベースの管理UIやダッシュボード
- データベースを使った記事の永続化
- 複雑なフィルタリングやレコメンデーションエンジン
- 複数Slackワークスペースへの同時配信
- 記事の詳細な分析やレポート生成

## 影響を受けるコンポーネント
- **新規作成**: 全システムコンポーネント（初期セットアップ）
  - ニュース収集モジュール
  - 要約生成モジュール
  - Slack通知モジュール
  - GitHub Actionsワークフロー
  - 設定ファイルとテストスイート

## 依存関係
- Ruby (最新安定版)
- Claude API (Anthropic) - 要約生成に必須
- Slack Webhook URL - 投稿配信に必須
- GitHub Actions - 実行環境として必須

## リスクと緩和策
| リスク | 影響 | 緩和策 |
|--------|------|--------|
| Claude API利用コスト超過 | 高 | トークン数制限、要約対象の絞り込み、キャッシング |
| ニュースソースのAPI変更 | 中 | エラーハンドリング、複数ソース対応、モジュール化 |
| GitHub Actions実行時間超過 | 中 | バッチサイズ制限、タイムアウト設定 |
| Slackレート制限 | 低 | 投稿間隔の調整、バッチング |

## 代替案
1. **手動キュレーション**: 自動化なし - 時間コストが高い
2. **既存サービス利用**: Feedly等 - カスタマイズ性が低い
3. **他のLLM利用**: GPT-4等 - Claude APIの品質と日本語対応を優先

## 成功基準
- [ ] システムが設定されたスケジュールで自動実行される
- [ ] 技術ニュースソースから記事を正常に収集できる
- [ ] Claude APIで記事を要約し、フォーマットされた出力を生成できる
- [ ] Slackに要約記事が正常に投稿される
- [ ] エラー発生時に適切なログが記録される
- [ ] テストスイートが全て通過する

## タイムライン
- フェーズ1 (基礎): プロジェクト構造、設定管理、ニュース収集 (2-3日)
- フェーズ2 (統合): 要約生成、Slack投稿、統合テスト (2-3日)
- フェーズ3 (自動化): GitHub Actions、エラーハンドリング、ドキュメント (1-2日)

合計見積もり: 5-8日

## 関連Capability

- `content-summarization`: LLMによる記事要約
- `slack-notification`: Slackへの投稿配信
- `scheduled-execution`: GitHub Actionsでの定期実行
